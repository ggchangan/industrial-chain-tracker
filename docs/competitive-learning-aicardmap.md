# AICardMap 学习记录与产品取舍

## 参照对象

- 站点：`https://aicardmap.com/?auth=google-ok`
- 观察日期：2026-06-27
- 参照目的：学习其信息分层、产品壳、权限机制和运营入口，不复制其定位。

## 对方的长处

AICardMap 的产品形态更像一个 AI 硬件产业链的导航数据库。它的层次感主要来自：

1. 固定应用壳  
   左侧主导航把总图、环节库、技术路线、公司库、情报雷达、学习路径和更新记录拆清楚。

2. 环节原子化  
   把 GPU、HBM、800G 光模块、液冷、变压器等拆成卡口节点，每个节点有上游、下游、瓶颈原因、相关公司和更新时间。

3. 总图优先  
   首页先给地图和关键节点，而不是先给长文章。

4. 两栏详情  
   主栏讲解释和判断，侧栏放基础信息、上下游、关系图、相关公司和更新时间。

5. 产品化入口完整  
   它有登录、会员订阅、非会员锁定、意见反馈、情报雷达和更新记录。这些能力让网站从内容页变成了可运营产品。

## 我们不应照搬的地方

我们的内容更新方式与它不同。

AICardMap 更像静态卡口百科：

```text
产业链大图 -> 卡口节点 -> 公司 / 技术路线 / 会员解锁
```

我们的核心应是持续更新某条产业链的逻辑：

```text
产业链地图 -> 逻辑地图 -> 证据更新 -> 逻辑强化 / 减弱 / 反证 -> 后续跟踪
```

因此，我们不应该只把每条产业链拆成一堆静态卡片。卡片可以作为入口，但真正的主线应该是：

- 这条产业链的核心判断是什么？
- 哪些逻辑正在发生变化？
- 新证据改变了哪条逻辑？
- 哪些指标或事件会验证、削弱或推翻它？
- 逻辑变化如何传导到环节、公司和市场表现？

## 我们应保留的自身长处

当前项目已有的优势：

- 多产业链深度研究原文。
- 每条产业链有图谱、骨架、核心逻辑和动态追踪。
- 标准研究包可以导入原文、HTML、logic.json、配图和监控项。
- 逻辑卡能关联公司、证据、监控项和原文锚点。
- 跟踪验证已经有“强化 / 稳定 / 减弱 / 反证 / 失效”的雏形。
- 个股集中营用于观察市场是否交易某条产业逻辑。

这些是研究系统的底层能力，不能为了像竞品那样简洁而丢掉。

## 值得学习的产品机制

### 1. 登录与会员

可以引入账号体系和会员权限，但权限边界要围绕我们的研究价值设计：

- 游客：可看首页、少量公开产业链、部分逻辑摘要和公开原文。
- 登录用户：可收藏产业链、订阅逻辑更新、保存阅读历史、提交反馈。
- 会员用户：解锁全部产业链、完整逻辑地图、历史逻辑变更、雷达订阅和高级资料归档。

非会员锁定不应简单粗暴遮住页面。更适合：

- 第一条产业链完整开放。
- 其他产业链开放地图摘要和少量核心逻辑。
- 深度逻辑、完整原文、历史证据链和雷达订阅需要会员。

### 2. 意见建议

需要一个轻量反馈入口，覆盖：

- 公司映射错误。
- 产业链环节遗漏。
- 来源链接失效。
- 逻辑判断异议。
- 希望新增产业链或研究主题。

反馈应能带上当前页面、产业链 ID、逻辑卡 ID 或公司名，方便维护者处理。

### 3. 情报雷达

我们可以做“产业链逻辑更新雷达”，不是简单新闻流。

雷达的核心对象应是逻辑和监控项：

```text
某条逻辑 -> 需要观察的信号 -> 新证据 -> 影响判断 -> 状态变化
```

示例：

- MPO 插芯缺货逻辑：订单、扩产周期、交期、毛利率。
- HBM 景气逻辑：价格、产能、客户认证、封测订单。
- CCL 价格传导逻辑：电子布、铜箔、树脂、覆铜板涨价函。

雷达页面不应只按时间列资讯，而应按逻辑分组：

- 今日强化的逻辑。
- 今日出现反证的逻辑。
- 待人工核验的证据。
- 已订阅产业链的关键变化。

### 4. 更新记录

需要保留公开更新记录，但重点不是“站点更新了什么”，而是：

- 哪条产业链新增了研究主题。
- 哪条逻辑被强化或削弱。
- 哪个证据被归档。
- 哪个监控项完成了核验。

## 建议的信息架构

### 全局层

```text
产业链研究库
├── 产业链地图
├── 逻辑地图
├── 环节库
├── 公司库
├── 逻辑雷达
├── 研究更新
└── 我的订阅
```

### 单条产业链层

```text
某产业链
├── 当前重点
├── 产业链地图
├── 逻辑地图
├── 最新证据
├── 跟踪验证
├── 公司与市场验证
└── 原文与资料档案
```

### 逻辑地图层

逻辑地图是我们区别于竞品的核心产品形态。

它应表达：

```text
需求变化
  -> 环节约束
    -> 公司受益 / 受损
      -> 证据验证
        -> 逻辑状态变化
```

每个逻辑节点至少包含：

- 逻辑标题。
- 当前状态：基准 / 强化 / 稳定 / 减弱 / 反证 / 失效。
- 影响环节。
- 相关公司。
- 关键证据。
- 待观察信号。
- 最近更新时间。
- 原文锚点。

### 产业链地图层

产业链地图负责结构，不负责承载所有判断。

它应表达：

- 上游、中游、下游。
- 关键环节。
- 供需传导。
- 价值量或瓶颈强度。
- 与逻辑节点的连接。

产业链地图回答“结构在哪里”，逻辑地图回答“判断如何变化”。

## 实施优先级

### 第一阶段：记录与展示层改造

- 增加“逻辑地图”概念和页面入口。
- 单条产业链首屏按“当前重点 / 产业链地图 / 逻辑地图 / 最新证据”重排。
- 把现有 logicTracks 渲染成逻辑地图雏形。
- 把 watchlist 和 trackingProfile 合并进逻辑节点的“待观察信号”。

### 第二阶段：雷达与订阅

- 新增逻辑雷达页面。
- 支持按产业链、逻辑状态、证据类型筛选。
- 登录用户可订阅产业链或逻辑。
- 新证据入库后生成雷达条目。

### 第三阶段：会员与权限

- 游客可看示例产业链和部分摘要。
- 登录用户可收藏、订阅、反馈。
- 会员解锁完整产业链、历史逻辑变更和高级雷达。
- 保持公开内容足够展示产品价值，避免纯遮挡式付费墙。

## 产品原则

1. 学习竞品的信息分层，不复制它的内容形态。
2. 产业链地图讲结构，逻辑地图讲判断，雷达讲变化。
3. 会员权限围绕深度研究和持续更新，不围绕简单数量限制。
4. 每次内容更新都要回答：它改变了哪条逻辑？
5. 原文是证据底座，逻辑地图是用户的第一阅读入口。
