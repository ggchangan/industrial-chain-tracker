# 逻辑监控执行策略

## 基本判断

产业链逻辑监控不是一套采集器覆盖全部指标，而是：

```text
通用运行框架 + 按数据源实现的专属采集器
```

通用框架负责定时任务、原始数据、变化计算、规则命中、候选证据、人工确认、
状态历史和告警。专属采集器负责数据在哪里、如何取得、字段口径和异常处理。

## 可执行性分级

### 自动监控

适合固定公开来源、结构化程度高、口径稳定的指标：

- 产品价格、价格指数
- 月度或季度产量、销量、出货量
- 库存、开工率、产能利用率
- 公司财报中的固定字段
- 交易所公告中的订单、中标、扩产和投产

程序可以自动采集和计算，但逻辑状态默认先进入待确认，避免异常数据直接改变研究判断。

### 半自动监控

程序能够发现候选资料，但难以稳定判断含义：

- 公告和财报正文中的业务描述
- 新闻、研报和机构观点
- 多家公司口径不一致的汇总指标

程序负责找证据、归档和提示可能命中的规则，维护者负责最终确认。

### 人工记录

数据来自产业调研、报价单、访谈、付费数据库或维护者已经掌握的资料。系统不要求用户
主动寻找证据，只负责记录、关联和展示。

### 观察项

没有稳定来源或当前成本过高的研究问题。保留其研究价值，但明确标记为“仅观察”，
不表现为正在运行的监控任务。

## 单个自动指标的最低配置

```text
监控对象
数据来源
采集频率
指标口径
强化条件
减弱条件
数据缺失和异常处理
自动判断还是人工确认
```

## 落地原则

1. 先选真实指标跑通，再抽象采集器接口。
2. 自动采集与逻辑判断分离。
3. 原始数据、计算结果和最终判断分别保存。
4. 不承诺所有逻辑都可自动监控，公开展示每项指标的可执行等级。
5. 优先建设可跨产业复用的数据源采集器，如交易所公告、财报字段和同站点价格序列。

## 后续试点

自动监控试点应同时满足来源稳定、指标口径明确、历史数据可验证。优先候选：

- 某一公开报价源上的产品价格
- 某一交易所公告中的订单或中标信息

北美云厂资本开支等跨公司、跨口径指标暂列为半自动或观察项，不作为首个自动化样板。
