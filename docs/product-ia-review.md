# 产业链研究库产品信息架构体检

## 背景判断

当前项目已经完成了研究包导入、资料归档、逻辑卡、监控核验、公司卡和个股集中营等能力。功能开始具备完整闭环，但页面信息密度也明显上升：

- 公开页模块变多，用户需要自己判断先看什么。
- 后台维护入口偏多，存在多个功能都能“新增资料 / 生成展示 / 影响公开页”的情况。
- 有些能力属于未来版本或高级维护，但现在和主流程并列展示。
- 小程序和 iOS App 不适合直接复制当前网页长页面，需要更清晰的移动端信息层级。

下一阶段重点不再继续深挖个股行情，而是让产业链研究本身更好读、更好查、更容易维护，并为 App 版本沉淀稳定的信息架构。

补充参照：

- `docs/competitive-learning-aicardmap.md` 记录了对 AICardMap 的学习取舍。结论是学习其导航壳、会员、反馈、雷达和更新记录等产品机制，但本项目的核心应继续围绕“产业链地图 + 逻辑地图 + 证据更新 + 跟踪验证”，而不是转成静态卡口百科。
- `docs/product-north-star-roadmap.md` 记录产品北极星和阶段路线图，用来约束后续功能优先级：先做样板产业链工作台，再做雷达、订阅、反馈和会员。

## 产品定位收敛

本版本核心目标：

```text
让用户快速理解一条产业链正在发生什么、核心逻辑是什么、证据在哪里、后续该跟踪什么。
```

暂缓目标：

- 项目内行情计算。
- 个股涨幅排序。
- 复杂自动监控。
- 面向所有资料类型的一键 AI 提炼。

这些方向有价值，但会把当前版本拖成行情系统或自动化监控系统。现阶段个股做到 K 线入口和产业链股票池即可。

## 公开页体检

### 当前模块

当前公开页大致是：

```text
产业链库 / 搜索
当前产业链 + 图谱
产业链骨架
逻辑监控
研究动态
个股集中营
核心逻辑
观察与验证
原文阅读
```

### 主要问题

1. 入口过多  
   导航里“逻辑监控 / 研究动态 / 个股集中营 / 核心逻辑 / 观察验证 / 原文阅读”全部并列，新用户不容易判断阅读顺序。

2. 逻辑相关模块分散  
   “逻辑监控”“核心逻辑”“观察与验证”“研究动态里的逻辑变化”都和逻辑有关，但散在不同位置。

3. 资料和逻辑之间的层级不够明确  
   用户导入 MPO 后，页面展示了资料归档、逻辑卡、监控项和动态，但没有一个明确的“这次研究更新了什么”的总览。

4. 原文阅读太重  
   原文阅读很重要，但在首页完整铺开会压低移动端阅读效率。移动端更适合把完整原文作为二级阅读页。

5. 个股集中营定位需要收敛  
   当前它很有价值，但应定位为“市场验证辅助入口”，不要在本版本继续扩展成行情分析中心。

## 公开页改版建议

建议改成 App 化信息结构：

```text
产业链详情
├── 当前重点
├── 最新研究
├── 核心逻辑
├── 产业链结构
├── 跟踪验证
├── 个股集中营
└── 原文阅读入口
```

### 1. 当前重点

放在产业链详情页顶部，回答：

- 这条产业链当前最重要的变化是什么？
- 最近新增了什么研究主题？
- 哪些逻辑正在强化或需要注意？

首版可以从现有数据自动拼：

- 最新研究包更新。
- 最新逻辑变化。
- 最新核验记录。
- 核心逻辑中的高优先级卡片。

展示不超过 3 条，避免变成另一个信息流。

### 2. 最新研究

合并现在的“研究动态”与“资料归档”入口，名字改为“最新研究”。

内部保留两个 tab：

- 逻辑变化
- 资料归档

但对用户展示时，默认先看“逻辑变化”，因为它更接近研究价值。

### 3. 核心逻辑

把“核心逻辑”拆成两层视觉结构：

```text
基准逻辑
近期研究逻辑
```

- 基准逻辑：原产业链长期有效的底层判断。
- 近期研究逻辑：由研究包 `logic.json` 生成的逻辑卡。

这样用户能理解 MPO 是“光模块产业链下的新研究主题”，而不是和基础产业链逻辑混在一起。

### 4. 产业链结构

合并“产业链图谱”和“产业链骨架”：

```text
产业链结构
├── 图谱
└── 上中下游骨架
```

移动端默认只展示图谱摘要和骨架卡片，完整大图可以点击查看。

### 5. 跟踪验证

合并“逻辑监控”和“观察与验证”。

建议结构：

```text
跟踪验证
├── 运行中的监控项
├── 待人工核验
└── 观察清单
```

旧的 watchlist 不必作为独立导航出现，可以降级为“观察清单”折叠区。

### 6. 个股集中营

保留，但文案收敛为：

```text
市场验证
```

它的任务是帮助用户打开 K 线，观察产业逻辑是否被市场交易。后续涨幅排序、强弱榜、区间收益放到下一版本。

### 7. 原文阅读

公开网页可以保留完整原文区；移动端建议改为入口：

```text
阅读完整研究原文
```

点击后进入独立阅读页，保留目录、进度、锚点和图片。

## 后台维护页体检

### 当前模块

后台当前大致是：

```text
导入标准研究包
产业链资料档案
人工核验监控项
新增逻辑卡
新增资料归档
更新原文
添加动态
资源概览
```

### 主要问题

1. 主流程和高级功能并列  
   “导入标准研究包”是当前最重要的流程，但“新增资料、逻辑卡、更新原文、添加动态”都在同级导航中，用户会犹豫应该走哪个入口。

2. 新增资料和添加动态重复  
   一份资料既可以作为归档，也可以生成动态。现在拆成两个表单，维护者需要重复填写来源标题、来源链接、产业链、日期等信息。

3. 逻辑卡手工表单过重  
   逻辑卡现在支持多种展示方式、结构化 JSON、排序、组 ID 等。这个能力强，但对日常维护来说太复杂，应降级为高级编辑。

4. 更新原文是低频操作  
   基准原文不是每天改，应该放到“高级维护”或“产业链设置”里。

5. 人工核验还不是主流程  
   当前监控体系仍在探索阶段，核验入口应保留，但不应和导入资料同权重。

## 后台维护改版建议

建议后台主导航改为：

```text
研究收件箱
资料档案
待办核验
高级维护
```

### 1. 研究收件箱

作为后台首页和主流程。

保留：

- 导入标准研究包
- 检测资料包
- 确认入库
- 入库后跳转到本次研究结果摘要

入库成功后，展示一张“导入结果卡”：

- 新增研究主题
- 新增 / 更新逻辑数
- 关联公司数
- 新增监控项数
- 原文入口
- 公开页入口

### 2. 资料档案

作为维护中心，而不是单纯列表。

每条资料卡应能看到：

- 原文
- HTML
- 图片
- logic.json
- 生成的逻辑卡
- 生成的动态
- 关联监控项

从资料档案发起：

- 编辑资料
- 提炼逻辑卡
- 发布动态
- 查看公开页呈现

### 3. 待办核验

替代当前“人工核验监控项”的大表单入口。

先列出待处理项：

- 哪条产业链
- 哪个监控项
- 为什么要核验
- 上次核验时间
- 推荐核验来源

用户点击某一项后再填写核验结果。这样比先选产业链再选监控项更符合任务心智。

### 4. 高级维护

收纳低频和复杂功能：

- 手工新增资料
- 手工新增 / 编辑逻辑卡
- 更新基准原文
- 添加动态
- 删除或隐藏内容
- 调整排序

这些功能保留，但默认不作为主流程打扰用户。

## 版本实施顺序

### 第一阶段：只改信息架构与文案

目标：减少重复感，不改变底层数据。

- 公开页导航重排和改名。
- 合并“逻辑监控 + 观察与验证”的展示入口。
- “研究动态”改名为“最新研究”。
- 后台导航简化为四个主入口。
- 把手工逻辑卡、更新原文、添加动态归到高级维护视觉分组。

验收标准：

- 用户进入公开页能在 10 秒内知道先看什么。
- 后台首页明确告诉用户“先导入研究包”。
- 不删除任何现有能力，只是降低低频功能的视觉权重。

### 第二阶段：研究导入结果摘要

目标：让研究包入库后价值立刻可见。

- 导入成功后显示导入摘要。
- 展示新增逻辑、公司、监控项、资料链接。
- 一键跳转公开页对应位置。

验收标准：

- 导入 MPO 后，不需要自己去页面各处找变化。
- 能从后台直接验证公开页呈现。

### 第三阶段：移动端详情页重构

目标：为小程序和 iOS App 统一内容体验。

- 移动端详情页按“当前重点 / 最新研究 / 核心逻辑 / 结构 / 跟踪 / 个股 / 原文”组织。
- 原文阅读独立化。
- 后端补充移动端友好的摘要字段。

验收标准：

- 小程序/App 不依赖网页长页结构。
- 移动端首页只显示重点信息，深度内容二级展开。

## 建议下一步

建议先做第一阶段：

```text
公开页和后台维护页的信息架构瘦身
```

这一步风险最小，收益最大。它不会改变数据结构，也不会影响研究包导入、COS、MySQL 或已有逻辑卡，只是让已经完成的能力更像一个产品。
